## 目标
在保持“整站尽量简单”的前提下，用清晰的目录边界支撑核心闭环：上传压缩包 → 安全解包 → 扫描 → 聚合判定 → 报告输出（JSON）→ 前端查看与下载。

## 目录结构（MVP）

## 目录职责

### scr/frontend（Web）
- pages：页面级路由与页面容器（上传页、任务列表、任务详情）
- components：可复用 UI 组件（上传器、状态标签、表格、明细列表等）
- services：与后端 API 的通信封装（tasks、reports）
- types：前端类型定义（Task、Finding、Report）
- utils：格式化、轮询、文件下载等通用工具
- tests：前端单测/组件测试

### scr/backend（Python）
- app：API 服务（上传/任务查询/结果与报告下载）
  - api：路由层（/api/tasks、/api/tasks/{id} 等）
  - core：配置、日志、版本（engine_version / ruleset_version）
  - db：数据库连接与迁移入口（MVP 可从 SQLite 起步）
  - models：持久化模型（Artifact/ScanTask/Finding）
  - schemas：API 入参与出参（Pydantic/类似方案）
  - services：业务服务（上传校验、安全解包、任务编排、报告生成）
- worker：异步执行入口（从队列取任务 → 调用 engine 扫描 → 写回结果）
- engine：扫描引擎（文件清单/分类 → 各类扫描器 → 聚合评分）
- rulesets：规则集与阈值配置（版本化；启停与阈值）
- storage：受控落盘目录与文件管理（artifact 存储、解包目录、临时目录）
- tests：后端与引擎测试（解包防护、规则命中、评分判定、报告结构）

### scr/shared（前后端共享）
- schemas：报告 JSON Schema、API 结构契约（用于对齐与校验）
- types：共享枚举与常量（Severity、Category、TaskStatus、Conclusion）

### scr/scripts（开发辅助）
- 本地运行、样例包生成、规则集验证、回归基线脚本等

## 关键文件命名建议（后续实现时落位）
- backend/rulesets/
  - ruleset.yaml（或 json）：规则启停、阈值、版本号
  - rules/：按类别拆分规则定义（secrets/dep/sast/config/malicious）
- backend/storage/
  - artifacts/{sha256}/original.(zip|tar.gz)
  - workdir/{task_id}/unpacked/
  - reports/{task_id}/report.json

## 代码流向（与目录对应）
- app/api → app/services（上传校验/创建任务）→ 入队 → worker → engine（扫描）→ 写回 db → app/api（查询/下载 report.json）

## 可扩展点（不影响 MVP 的前提下预留）
- engine/scanners：按类别扩展扫描器，实现渐进新增规则
- rulesets：支持多版本规则集并在报告中记录 ruleset_version，确保结果可复现
